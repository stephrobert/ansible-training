# Lab 86 — Build your own EE with ansible-builder

> 💡 **Prerequisites**:
> - **Podman** installed (`podman --version`).
> - **ansible-builder** ≥ 3.1.0: `pipx install ansible-builder`.
> - Optional: **ansible-navigator** to test the built EE.

## 🧠 Recap

🔗 [**Build an EE with ansible-builder**](https://blog.stephane-robert.info/docs/infra-as-code/gestion-de-configuration/ansible/execution-environments/ee-builder/)

**`ansible-builder`** v3 reads an **`execution-environment.yml`** v3 file and generates a multi-stage **`Containerfile`** that produces the final OCI image via **Podman**. Four files coexist:

- **`execution-environment.yml`**: recipe (version, base image, deps, build steps).
- **`requirements.yml`**: Ansible collections (Galaxy).
- **`requirements.txt`**: Python dependencies (pip).
- **`bindep.txt`**: system dependencies (dnf/apt depending on the base).

## 🎯 Objectives

By the end of this lab, you will know how to:

1. Write an **`execution-environment.yml`** in the v3 schema.
2. Pin the **versions**: ansible-core, collections, Python deps, system deps.
3. Run **`ansible-builder build`** with Podman.
4. Inspect the **`Containerfile`** generated in `./context/`.
5. Test the built EE with **`ansible-navigator`**.
6. Push the EE to a private registry (Quay, Harbor, GitLab Registry).

## 📚 Exercise 1 — Inspect execution-environment.yml v3

```yaml
version: 3                                    # ← MANDATORY in 2026

images:
  base_image:
    name: ghcr.io/ansible-community/community-ee-minimal:latest

dependencies:
  ansible_core:
    package_pip: ansible-core==2.18.1         # strict pinning
  ansible_runner:
    package_pip: ansible-runner==2.4.1
  galaxy: requirements.yml                    # collections
  python: requirements.txt                    # pip
  system: bindep.txt                          # dnf

additional_build_files:
  - src: configs/ansible.cfg
    dest: configs

additional_build_steps:
  prepend_base:
    - RUN echo "Build $(date)" > /etc/ee-build-info
  append_final:
    - COPY _build/configs/ansible.cfg /etc/ansible/ansible.cfg
```

🔍 **Observation**: `version: 3` is **mandatory**. Without it, ansible-builder reads in legacy v1 mode and **ignores** `dependencies.ansible_core`: a classic source of bugs.

## 📚 Exercise 2 — Build with Podman

```bash
ansible-builder build \
  --tag localhost/lab86-my-ee:1.0.0 \
  --container-runtime podman \
  --file execution-environment.yml \
  --context ./context \
  --verbosity 2
```

Typical output:

```text
[1/3] Resolving collections via Galaxy...
[2/3] Building intermediate image (galaxy stage)...
[3/3] Building final image...
Successfully tagged localhost/lab86-my-ee:1.0.0
```

🔍 **Observation**: ansible-builder generates a multi-stage **`./context/Containerfile`** (4 stages: base, galaxy, builder, final). Inspect it to understand.

## 📚 Exercise 3 — Verify the built EE

```bash
podman images | grep lab86

# embedded ansible-core
podman run --rm localhost/lab86-my-ee:1.0.0 ansible --version | head -1
# → ansible [core 2.18.1]

# embedded collections
podman run --rm localhost/lab86-my-ee:1.0.0 ansible-galaxy collection list
```

Expected output:

```text
ansible.builtin            2.18.1
ansible.posix              2.0.0
community.general          10.5.0
kubernetes.core            5.1.1
```

🔍 **Observation**: only the collections **explicitly declared** in `requirements.yml` (+ the automatic `ansible.builtin`) are present. No bloat: that is the benefit of starting from a `community-ee-minimal`.

## 📚 Exercise 4 — Test the EE with ansible-navigator

```bash
# Create a minimal playbook
cat > test.yml <<'EOF'
---
- hosts: localhost
  gather_facts: false
  tasks:
    - name: Vérifier kubernetes.core
      ansible.builtin.debug:
        msg: "kubernetes.core OK"
EOF

# Run inside the custom EE
ansible-navigator run test.yml \
  --eei localhost/lab86-my-ee:1.0.0 \
  -m stdout
```

🔍 **Observation**: navigator launches Podman with your custom EE, not with creator-ee. You **control** what runs, **line by line** in `execution-environment.yml`.

## 📚 Exercise 5 — Rigorous pinning and reproducibility

Compare:

| Practice | Effect |
|----------|-------|
| `ansible-core: ansible-core` | Fetches the latest at build time -> not reproducible. |
| `ansible-core==2.18.1` | Always 2.18.1. **Reproducible**. |
| `community.general` (without version) | Latest at build time. |
| `community.general 10.5.0` | Exact version. **Recommended**. |

🔍 **Crucial observation**: **in 2026, we pin everything**. An image rebuilt in 6 months must give **bit-for-bit the same result**, otherwise the "freeze the runtime" benefit is cancelled.

## 📚 Exercise 6 — Push to a private registry

```bash
# Tag for Quay
podman tag localhost/lab86-my-ee:1.0.0 quay.io/myorg/my-ee:1.0.0

# Login
podman login quay.io

# Push
podman push quay.io/myorg/my-ee:1.0.0
```

🔍 **Observation**: semver pattern, always push `:1.0.0` (immutable patch tag), then `:1.0` (minor), `:1` (major), and a `:latest` that follows the last stable. **Never overwrite** a published patch tag.

## 🔍 Observations to note

- **Idempotence**: a second run of your solution must show `changed=0`
  everywhere in the `PLAY RECAP`. This is the mechanical signal of a playbook
  compliant with best practices.
- **Explicit FQCN**: always prefer `ansible.builtin.<module>` (or the
  appropriate collection) over the short name. `ansible-lint --profile
  production` checks it.
- **Targeting convention**: this lab targets your local machine + a custom OCI EE. To adapt it to
  another group, adjust `hosts:` in `lab.yml`/`solution.yml` then rerun.
- **Isolated reset**: `dsoxlab clean <lab-id>` at the lab root cleanly uninstalls
  what the solution set up so you can replay the scenario.

## 🔍 Security — 2026 best practices

- **Strict pin** on **all** the layers (ansible-core, collections, Python, system).
- **Base image pinned by digest**: `community-ee-minimal@sha256:abc...` (not `:latest`).
- **Trivy scan** in CI: `trivy image --severity HIGH,CRITICAL --exit-code 1 my-ee:1.0.0`.
- **SBOM**: `syft my-ee:1.0.0 -o spdx-json > sbom.json`.
- **Cosign sign** before pushing to production: `cosign sign quay.io/myorg/my-ee:1.0.0`.
- **Private registry** + auth (no public `:latest` in production).
