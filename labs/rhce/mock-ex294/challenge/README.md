# Challenge ŌĆö Mock RHCE EX294: 19 tasks in 4 hours

## Exam conditions

- **Duration**: 4 hours on the clock.
- **Environment**: 4 VMs (`web1.lab`, `web2.lab`, `db1.lab`, `control-node.lab`).
- **Expected knowledge**: choice of the FQCN module, parameters, idempotence.
- **No external help**: offline Ansible doc (`ansible-doc`) allowed. No
  Internet other than the Red Hat Ansible doc.
- **Success criterion**: the pytest tests green (19 tasks, several
  of them checked by more than one test).

## Prerequisites (to validate before starting the clock)

```bash
cd $ANSIBLE_TRAINING
ansible all -m ansible.builtin.ping              # 4 'pong' attendus
```

The exam environment is **set** for you: as in the real Red Hat exam,
the room is ready and you do not have to build it.

- VG `vg_lab` present on `db1.lab` (T10).
- UID `2001` free on all the hosts (T7).
- Port 80 free on the `webservers` (T5/T6).
- `semanage` available on the `webservers` (verification of T8).

It is the lab's `setup.yaml` that sets this state, and it sets it **by undoing**
the expected result of the 18 tasks: SELinux boolean to `off`, firewalld services
closed, `nginx` and `mariadb-server` uninstalled, `appuser` removed, crontab
and custom facts removed, markers erased. Nothing is given to you by
a previous run.

```bash
dsoxlab run rhce-mock-ex294       # plays the setup.yaml, then hands control back to you
```

Start the clock **after**.

## The 19 tasks

### Task 1 ŌĆö Static inventory

Produce `inventory/hosts.yml` in YAML format declaring:

- Group `webservers`: `web1.lab`, `web2.lab`.
- Group `dbservers`: `db1.lab`.
- Global variables for all the hosts:
  - `ansible_user: ansible` (the service account that dsoxlab provisions on the VMs).
  - `ansible_python_interpreter: /usr/bin/python3`
  - enough to reach the VMs without coding a single IP: Terraform assigns
    them and they change at each reprovisioning. The ssh_config generated by
    dsoxlab (`~/.cache/dsoxlab/ansible-training/ssh_config`) carries the
    real addresses and the key; pass it to OpenSSH via
    `ansible_ssh_common_args: -F <path>`.

`ansible-inventory -i inventory/hosts.yml --graph` must display the two
groups with their hosts.

### Task 2 ŌĆö Hierarchical variables

Define:

| File | Variable | Value |
| --- | --- | --- |
| `inventory/group_vars/all.yml` | `app_env` | `production` |
| `inventory/group_vars/webservers.yml` | `worker_count` | `4` |
| `inventory/host_vars/db1.lab.yml` | `db_name` | `lab100db` |

On `web1.lab`, deposit the file `/tmp/lab100-vars.txt` (mode `0644`) that
contains at minimum the lines:

```text
app_env: production
worker_count: 4
```

### Task 3 ŌĆö Vault

- Create `.vault_password` (mode `0600`) with a password of your choice.
- Create `vault.yml`, **encrypted**, containing the variable:
  - `db_password: "Lab92Pass!"`
- Load this file in the playbook via `vars_files:`.
- On `db1.lab`, write `/tmp/lab100-vault-test.txt` (mode `0600`) containing
  the decrypted value of `db_password`.

### Task 4 ŌĆö Template

On `db1.lab`, deposit `/tmp/lab100-app.conf`:

- mode `0640`, owner `appuser`, group `appgroup` (see task 7).
- content generated from a Jinja2 template including **at minimum** the Ansible
  hostname and the value of `db_name` (the string `lab100db` must appear).

### Task 5 ŌĆö Packages

Install on **all** the hosts: `nginx`, `mariadb-server`, `python3-libselinux`.

### Task 6 ŌĆö Services

- On `webservers`: `nginx` enabled at boot and started.
- On `db1.lab`: `mariadb` enabled at boot and started.

### Task 7 ŌĆö Users and groups

On **all** the hosts:

- Group `appgroup` (GID `2001`).
- User `appuser` (UID `2001`, primary group `appgroup`, shell `/bin/bash`).
- The user `appuser` must be able to run `/usr/bin/systemctl` via `sudo` **without
  entering a password** (only this command).

### Task 8 ŌĆö SELinux

On `webservers`, enable the SELinux boolean `httpd_can_network_connect`
**permanently** (survives the reboot).

> ­¤ÆĪ The boolean is named `httpd_*` while the installed web server is
> nginx: this is not a statement error. On RHEL, nginx runs in the
> **SELinux domain `httpd_t`**, which is the generic domain of web servers
> in the targeted policy. Check it yourself on web1:
> `ps -eZ | grep nginx` displays `system_u:system_r:httpd_t:s0`. The booleans
> `httpd_can_network_connect`, `httpd_enable_homedirs` and the like therefore
> apply to nginx.

### Task 9 ŌĆö Firewalld

- On `webservers`: open the services `http` and `https`.
- On `db1.lab`: open the service `mysql`.

The rules must be **persistent** (reboot) **and active immediately**
(no need for `firewall-cmd --reload` after the playbook).

### Task 10 ŌĆö LVM storage

On `db1.lab`, in the existing VG `vg_lab`:

- Create an LV `lv_data` of **300 MB**.
- Format it in **XFS**.
- Mount it on `/mnt/data`, with an `fstab` entry that **persists at reboot**.

### Task 11 ŌĆö Role `app_deploy`

Create a role `app_deploy` (under `roles/app_deploy/`) that, on the `webservers`:

- Deposits `/usr/share/nginx/html/index.html` from a Jinja2 template containing
  at least the inventory hostname.
- Guarantees that `nginx` is active (idempotent, task 6 has already done it,
  your role must not break).
- Defines a handler `Restart nginx` notified if the template changes.

The `solution.yml` invokes this role for the `webservers` (e.g. `roles:` or
`include_role:` in a `play` that targets `webservers`).

### Task 12 ŌĆö Loops and conditions

On `db1.lab`, create 5 files `/tmp/file1` to `/tmp/file5`:

- **even** number (2, 4): content `pair`.
- **odd** number (1, 3, 5): content `impair`.

The 5 files must be created in **a single task** with a loop.

### Task 13 ŌĆö Incident handling

On `db1.lab`, the playbook must attempt to start the service
`lab100-collecteur`. No package of the distribution provides this service: the
attempt **really fails**, on every pass. This is intended.

- The task that fails is named **exactly** `D├®marrer le collecteur lab100`.
- The failure must **not** interrupt the playbook. For `db1.lab`, the
  `PLAY RECAP` must display `failed=0`.
- The incident must be **caught**, not passed over in silence: the `PLAY RECAP`
  must display `rescued=1` and `ignored=0`. Silencing the error is rejected.
- **Only in case of failure**: write `/tmp/lab100-incident.txt` (mode
  `0644`) containing, on a line of its own, the name of the task that failed **as
  Ansible reports it**. This name is read at runtime, it is not copied out.
- **In all cases**, success as well as failure: write
  `/tmp/lab100-incident-fin.txt` (mode `0644`).

### Task 14 ŌĆö Rolling deployment

On the `webservers`, the deployment must be done **in waves of a single host at
a time**, in inventory order (`web1.lab`, then `web2.lab`).

- Each host deposits `/tmp/lab100-vague-<inventory_hostname>.txt` (mode `0644`,
  owner `root`). Example: `/tmp/lab100-vague-web1.lab.txt`.
- The content of the marker must be **stable from one pass to the next** (a
  timestamp in the content would break the idempotence required below).
- Between two waves, the deployment must observe a **stabilization time
  of at least 5 seconds**, the time for a freshly deployed host to settle before
  attacking the next one.

> The verification is based on the **timestamps** (`mtime`) of the two markers:
> that of `web1.lab` must precede that of `web2.lab` by at least 3 seconds.
> A parallel deployment deposits them a few milliseconds apart.

### Task 15 ŌĆö Centralized deployment log

From a play that targets the **`webservers`**, write the log
`/tmp/lab100-deploy-log.txt` (mode `0644`, owner `root`) **on `db1.lab`**.

- It must contain **exactly one line**, whatever the number of
  webservers, and replaying the playbook must not add any.
- This line must name the **webserver** at the origin of the write (its
  `inventory_hostname`) and carry the value of `worker_count` (see task 2).
- The file must exist **on no** webserver.

### Task 16 ŌĆö Scheduled task

On `db1.lab`, schedule a report in the **crontab of the user
`appuser`** (see task 7), that is the one that
`crontab -l -u appuser` returns:

- schedule: every day at **04h05**;
- command: `/usr/bin/date >> /tmp/lab100-rapport.log 2>&1`;
- one entry and only one: replaying the playbook must not duplicate it.

### Task 17 ŌĆö Task selection by tags

On `db1.lab`, four tasks, each depositing its marker in mode `0644`:

| Task | Marker | Must run |
| --- | --- | --- |
| Preparation marker | `/tmp/lab100-tag-preparation.txt` | under the tag `preparation` |
| Deployment marker | `/tmp/lab100-tag-deploiement.txt` | under the tag `deploiement` |
| Systematic marker | `/tmp/lab100-tag-always.txt` | **always**, including under `--tags deploiement` |
| Destructive purge | `/tmp/lab100-tag-purge.txt` | **never**, except under `--tags purge` requested explicitly |

The purge, in addition to depositing its marker, removes those of preparation and
deployment.

The two behaviors verified, on real runs:

- **Without `--tags`**: preparation, deployment and systematic are deposited; the
  purge did **not** happen.
- **With `--tags deploiement`**: deployment and systematic are deposited;
  preparation and purge are **absent**.

### Task 18 ŌĆö Custom fact

On the `webservers`, publish a custom fact that any fact collection
will find:

- directory `/etc/ansible/facts.d` (mode `0755`, owner `root`);
- file `lab100.fact` (mode `0644`, owner `root`), in **INI** format, section
  `[exam]`, with two keys:
  - `env`: the value of `app_env` (see task 2);
  - `workers`: the value of `worker_count` (see task 2).

The values must **come from the inventory**, not be copied in hard.

The verification does not open the file: it asks Ansible what it
retrieves from it. The following command must return `ansible_local.lab100.exam.env` and
`ansible_local.lab100.exam.workers` on **both** webservers:

```bash
ansible webservers -i inventory/hosts.yml \
    -m ansible.builtin.setup -a filter=ansible_local
```

### Task 19 ŌĆö Content Collection via `requirements.yml`

The EX294 objective **"Install Content Collections and use them in playbooks"**
is not proven by any of the previous tasks: add it.

Write a **`requirements.yml`** declaring **one** collection from Ansible Galaxy,
its version **pinned in strict semver** (`community.general` version `10.5.0`).
Install it into a **project-local directory** (not the home), with the install
step made **idempotent** (`creates:` on `ansible-galaxy`, which is a child
process and knows nothing of a second pass).

Then, from `db1.lab`:

- Deposit `/tmp/lab100-collections.txt` (mode `0644`, owner `root`) containing
  the output of `ansible-galaxy collection list` for that install path. It must
  name the collection **and** its pinned version, the proof that it is really
  installed and resolvable.
- **Use** a module **from the installed collection**: `community.general.ini_file`
  writes `/tmp/lab100-collection-use.ini` (mode `0644`, owner `root`) with the
  section `[collections]` and the key `installed = community.general`.

> The verification does not read your `requirements.yml`: it checks the state it
> produced. A `requirements.yml` that declares the right sources but is never
> installed leaves no `10.5.0` in the collection list, and a file obtained by a
> plain `copy` does not prove you can use a collection module.

## Validation

```bash
pytest -v labs/rhce/mock-ex294/challenge/tests/
```

The success scale is in the root README of the lab. The 19 tasks are
covered by the pytest suite: several tasks have two or three tests, because an
RHCE task is rarely failed as a whole. You can open a port without making it
permanent, enable a SELinux boolean that drops back at reboot, mount a volume
without an fstab entry, or grant `appuser` a sudo much broader than requested:
each time the state is right at time T and wrong after reboot, or right and
dangerous. These are exactly the points that the real exam sanctions.

Tasks 13 to 18 are verified the same way, on the real state of the
machines: the counters of the `PLAY RECAP` for the caught incident, the
timestamps of the markers for the waves, the number of lines of the log for
the delegation, `crontab -l -u appuser` for the scheduled task, a real
`--tags` run for the selection, and a fact collection for `ansible_local`.
Obtaining the right file by the wrong means does not pass: an error ignored
instead of being caught leaves the same file behind it, and the `PLAY RECAP`
denounces it.

## ­¤ÆĪ Going further

- **`ansible-lint --profile production`**: validate the quality of your solution.

  ```bash
  ansible-lint --profile production labs/rhce/mock-ex294/challenge/solution.yml
  ```

  Expected output: `Passed: 0 failure(s), 0 warning(s)`.

- **Idempotence**: rerun the solution a second time. A `PLAY RECAP`
  with `changed=0` everywhere confirms a clean playbook. It is an exam
  criterion, and a test checks it.

- **Task 14, proof by absurdity**: remove the keyword that sequences the
  waves and rerun. The two markers are deposited at the same time, and the test
  turns red. That is what makes the difference between a test that measures and a
  test that hopes.

- **Edge cases**: beyond task 13, think of the other error scenarios
  (unavailable host, missing dependency, invalid value) that your solution
  could encounter in production. `failed_when`, `assert` and
  `any_errors_fatal` complement `block`/`rescue`.
