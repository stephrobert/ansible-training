## 🧠 Recap

🔗 [**Role → collection migration**](https://blog.stephane-robert.info/docs/infra-as-code/gestion-de-configuration/ansible/collections/migration-role/)

In 2026, **standalone Galaxy v1 roles are legacy**. Galaxy NG (Galaxy v3 backend) supports **only** collections. If you maintain a public role, you must **migrate it to a collection** without breaking the downstream playbooks that still use it with its old name.

Official mechanism: **`meta/runtime.yml`** with **`plugin_routing.redirect`**. A playbook that invokes `webapp_module` keeps working (with a **deprecation warning**) while the real code now lives in `mycoll.webapp.webapp_module`.

This lab practices this migration on a mini-role, and **proves** that both names (old standalone + new FQCN) work.

## 🎯 Objectives

By the end of this lab, you will know how to:

1. **Identify** a standalone role that is a migration candidate.
2. **Initialize** the target collection with `ansible-galaxy collection init`.
3. **Move** `tasks/`, `handlers/`, `defaults/`, `templates/` to `roles/<name>/`.
4. **Move** a custom module from `library/` to `plugins/modules/`.
5. **Configure `plugin_routing.redirect`** in `meta/runtime.yml`.
6. **Test** that the old name triggers a deprecation warning **but works**.
7. **Test** that the new FQCN is the recommended target.

## 📚 Exercise 1 — The starting point: a legacy role

Create the original standalone role:

```bash
mkdir -p labs/collections/migration-role/roles_legacy/webapp/{tasks,library}
```

`labs/collections/migration-role/roles_legacy/webapp/tasks/main.yml`:

```yaml
---
- name: Status check via le module legacy lab97_status
  lab97_status:        # ← short name (no FQCN), legacy
  register: status

- ansible.builtin.debug:
    msg: "Statut: {{ status.msg }}"
```

`labs/collections/migration-role/roles_legacy/webapp/library/lab97_status.py`:

```python
#!/usr/bin/python
from ansible.module_utils.basic import AnsibleModule
def main():
    AnsibleModule(argument_spec={}).exit_json(
        changed=False, msg="lab97-legacy-OK"
    )
if __name__ == "__main__":
    main()
```

🔍 **Observation**: this is the **historical format** of standalone roles: `tasks/main.yml` + a `library/` folder that contains the custom modules. Galaxy v1 accepted this. Galaxy NG does **not**.

ansible-galaxy collection init student.webapp \

## 📚 Exercise 3 — Migrate the role into `roles/`

```bash
mkdir -p ansible_collections/student/webapp/roles/webapp/tasks
cp roles_legacy/webapp/tasks/main.yml \
   ansible_collections/student/webapp/roles/webapp/tasks/main.yml
```

Adapt `tasks/main.yml` to use the **new FQCN**:

```yaml
---
- name: Status check via le module migré
  student.webapp.lab97_check:     # ← new FQCN, new name
  register: status

- ansible.builtin.debug:
    msg: "Statut: {{ status.msg }}"
```

🔍 **Observation**: **inside** the collection, you can write short names (`lab97_check:`), but this is **discouraged**. The explicit FQCN stays **readable** and **portable**.

## 📚 Exercise 4 — Migrate the module into `plugins/modules/`

```bash
mkdir -p ansible_collections/student/webapp/plugins/modules
cp roles_legacy/webapp/library/lab97_status.py \
   ansible_collections/student/webapp/plugins/modules/lab97_check.py
```

Complete the module with the mandatory sections `DOCUMENTATION`, `EXAMPLES`, `RETURN`:

```python
#!/usr/bin/python
# -*- coding: utf-8 -*-

from __future__ import absolute_import, division, print_function
__metaclass__ = type

DOCUMENTATION = r'''
---
module: lab97_check
short_description: Module de démonstration migré du rôle standalone vers une collection
version_added: "1.0.0"
description:
  - Retourne un statut OK pour valider la migration.
options: {}
author:
  - "Apprenant RHCE 2026"
'''

EXAMPLES = r'''
- name: Status check
  student.webapp.lab97_check:
'''

RETURN = r'''
msg:
  description: Message de statut.
  type: str
  returned: always
'''

from ansible.module_utils.basic import AnsibleModule

def main():
    AnsibleModule(argument_spec={}).exit_json(changed=False, msg="lab97-migrated-OK")

if __name__ == "__main__":
    main()
```

🔍 **Observation**: `plugins/modules/<name>.py` is the **standard location**. The legacy `library/` at the root of the role is **no longer recognized** as soon as a role lives in a collection.

## 📚 Exercise 5 — Configure `plugin_routing.redirect` in `meta/runtime.yml`

This is **the heart** of the migration: letting a playbook that invokes the **old short name** `lab97_status:` keep working.

Along the way, we take the opportunity to **normalize the name** of the module: `lab97_status`
in the standalone role becomes `lab97_check` in the collection. It is the
redirect that keeps the old name alive.

`ansible_collections/student/webapp/meta/runtime.yml`:

```yaml
---
requires_ansible: ">=2.18.0"

plugin_routing:
  modules:
    lab97_status:                       # ← the OLD name, the one we keep alive
      redirect: student.webapp.lab97_check   # ← the REAL name in the collection
      deprecation:
        removal_version: 2.0.0
        warning_text: |
          Le module 'lab97_status' (rôle standalone) est déprécié.
          Utilisez 'student.webapp.lab97_check' (FQCN collection).
```

🔍 **Crucial observation**: **`plugin_routing.modules.<old_name>.redirect`** declares the target. **`deprecation:`** adds a **warning** at runtime but **does not fail**. **`removal_version`** indicates when the alias will be removed (semver of the collection). Essential pattern for backward compatibility.

⚠️ **The key and the target must differ.** Writing
`lab97_status: {redirect: student.webapp.lab97_status}` in the collection
`student.webapp` means redirecting `student.webapp.lab97_status` to
itself. ansible-core refuses at runtime:

```text
[ERROR]: plugin redirect loop resolving lab97_status
         (path: ['lab97_status', 'student.webapp.lab97_status'])
```

`plugin_routing` routes a **name** to **another name**: a rename, or a
move to another collection. It does not "move" a module to
where it already is. A module simply taken out of the `library/` of a
role **without changing its name** needs no route: the `collections:`
keyword of the playbook, or its FQCN, is enough to resolve it.

## 📚 Exercise 6 — Test the old name + the new FQCN

Create a test playbook:

```yaml
---
- hosts: db1.lab
  collections:
    - student.webapp           # ← declares only the collection (backward compat)
  tasks:
    - name: Test ancien nom court (avec deprecation warning)
      lab97_status:             # ← the old name, which will be redirected
      register: legacy

    - name: Test nouveau FQCN explicite
      student.webapp.lab97_check:
      register: new

    - ansible.builtin.copy:
        dest: /tmp/lab97-migration.txt
        content: |
          legacy: {{ legacy.msg }}
          new: {{ new.msg }}
        mode: "0644"
```

Run:

```bash
ANSIBLE_COLLECTIONS_PATH=labs/collections/migration-role/ansible_collections \
  ansible-playbook lab.yml
```

Expected output:

```text
[DEPRECATION WARNING]: student.webapp.lab97_status has been deprecated. Le module
'lab97_status' (rôle standalone) est déprécié...
TASK [Test ancien nom court (avec deprecation warning)] ***
ok: [db1.lab]
TASK [Test nouveau FQCN explicite] ***
ok: [db1.lab]
```

🔍 **Observation**: **both names work**. The old name triggers a **warning** but does not break. Downstream playbooks can migrate **progressively** to the FQCN.

## 🔍 Observations to note

- **Standalone roles** = legacy. Galaxy NG supports only collections.
- **Migration** = `tasks/`, `handlers/`, `defaults/` → `roles/<name>/`. `library/` → `plugins/modules/`.
- **`plugin_routing.redirect`** in `meta/runtime.yml` enables backward compatibility.
- **The key of the route and its target must differ**: redirecting a name to itself is a loop, refused at runtime.
- **`deprecation:`** adds a warning at runtime + `removal_version` schedules the removal.
- **The old name** keeps working: downstream migrate progressively.
