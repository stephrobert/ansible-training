## 🧠 Recap

🔗 [**Dynamic inventory for KVM**](https://blog.stephane-robert.info/docs/infra-as-code/gestion-de-configuration/ansible/inventaires/dynamiques/plugin-libvirt-kvm/)

The **`community.libvirt.libvirt`** plugin queries the local **libvirt API** and generates an inventory **on every run** of Ansible. No more manually maintaining the list of hosts: create a VM, and it appears immediately in the inventory.

**Limitation**: the plugin returns the libvirt **domain name** and its **state**, but **no usable IP**. It does expose `interface_addresses` and `guest_info`, but these two variables require the **QEMU agent** (`qemu-guest-agent`) installed in the VM and the `org.qemu.guest_agent.0` channel configured. That is not the case here: both contain only an error.

```bash
ansible-inventory -i inventory/ --host web1.lab 2>/dev/null | grep -A2 interface_addresses
```

```json
"interface_addresses": {
    "error": "argument unsupported: QEMU guest agent is not configured"
},
```

For the SSH connection, we therefore combine the plugin with **static variables** that say **how** to reach the VMs. Careful: **not by writing IPs in them**. Terraform assigns them and changes them at each reprovisioning; an IP frozen in an inventory is a wrong IP by the next day. We delegate resolution to the **`ssh_config` generated by dsoxlab**, which carries the real addresses, the user and the key.

## 📚 Exercise 1: configure the libvirt plugin

The file `inventory/01-libvirt.yml`:

```yaml
---
plugin: community.libvirt.libvirt
uri: qemu:///system
inventory_hostname: name
strict: false

groups:
  lab_vms: inventory_hostname in ['control-node.lab', 'web1.lab', 'web2.lab', 'db1.lab']
  webservers: inventory_hostname in ['web1.lab', 'web2.lab']
  dbservers: inventory_hostname == 'db1.lab'
  control: inventory_hostname == 'control-node.lab'

keyed_groups:
  - prefix: state
    key: info.state
```

🔍 **Observation**:

- **`plugin:`** at the top level tells Ansible that this file is a plugin config (not a classic YAML inventory).
- **`uri:`** points to the libvirt socket: `qemu:///system` for root KVM, `qemu:///session` for user-mode.
- **`inventory_hostname: name`** makes the host carry the **libvirt domain name**, the one `virsh list --all` shows. Here the domains are called `web1.lab`, `db1.lab`, and so on: it is this name, and this name only, that the filters see. A filter written on `web1` would match **nothing**, silently.
- **`groups:`** creates logical groups from a Jinja condition evaluated **per VM**.
- **`keyed_groups:`** creates one group per **value** of a variable: `state_running`, `state_shutoff`, and so on.
- **`strict: false`** (the default): a condition that references an absent variable drops the host from the group instead of failing the whole run. This is what makes the inventory tolerant to the VMs that libvirt refuses to detail ("domain is not running").

⚠️ **The `keyed_groups` trap**: the key is **`info.state`**, not `state`. The plugin stores the domain state in an `info` dictionary. With `key: state`, the variable does not exist, `strict: false` swallows the error, the `state_running` group is **never created**, and a play targeting `lab_vms:&state_running` matches no one. Ansible then exits with `rc=0` on a `skipping: no hosts matched`: **a perfectly silent failure**.

## 📚 Exercise 2: list the VMs

```bash
ansible-inventory -i inventory/01-libvirt.yml --graph 2>/dev/null
```

You see **all the VMs** of the system: the lab ones plus all the others (your personal VMs, those of other trainings, and so on). This is the expected behavior of the plugin: it queries libvirt, libvirt returns everything.

```bash
ansible-inventory -i inventory/ --graph lab_vms 2>/dev/null
```

The `lab_vms` group contains only the **4 lab VMs** (Jinja filter):

```text
@lab_vms:
  |--control-node.lab
  |--db1.lab
  |--web1.lab
  |--web2.lab
```

This is **the** point that makes this lab safe: `lab_vms` is an **explicit allowlist**. A playbook targeting `all` would go and write on your personal machines.

## 📚 Exercise 3: inspect the variables of a VM

```bash
ansible-inventory -i inventory/ --host web1.lab 2>/dev/null | head -20
```

The plugin returns facts about each VM:

- **`info`**: the state (`info.state` → `running`/`shutoff`), the memory, the vCPUs, the CPU time.
- **`xml_desc`**: the complete libvirt XML config (network, disks, UUID).
- **`guest_info`** and **`interface_addresses`**: CPU, OS, IP, and so on, **if the QEMU agent is installed**. Here, an error.

You can use them in `when:` conditionals, `groups:`, or templates.

## 📚 Exercise 4: add the SSH parameters (and understand the precedence)

The plugin does **not know how to connect over SSH**. Worse, it imposes its own connection mode:

```bash
ansible-inventory -i inventory/01-libvirt.yml --host web1.lab 2>/dev/null | grep ansible_connection
```

```json
"ansible_connection": "community.libvirt.libvirt_qemu",
```

The plugin **forces `ansible_connection: community.libvirt.libvirt_qemu`** on each host: it wants to go through the libvirt API and the QEMU agent. Since the agent is not there, we have to switch back to classic SSH. This is where the **variable precedence** decides everything.

Reminder of the Ansible order (from weakest to strongest):

| Rank | Source |
| --- | --- |
| 5 | inventory **group** variables (`group_vars/`) |
| 7 | **host** variables set by the **inventory** (← the plugin is here) |
| 8 | **host** variables in a file (`host_vars/`) |

Direct consequence, and this is **the** lesson of the lab:

- `ansible_connection` must be set in **`host_vars/`** (rank 8) to beat the plugin (rank 7). In `group_vars/` (rank 5), it would **lose**, without a word of warning.
- Everything else (`ansible_user`, the key, the `ssh_config`) is set by no one else: the **group** is enough, and it avoids repeating the same lines four times.

`inventory/host_vars/web1.lab.yml` therefore contains only one useful line:

```yaml
---
ansible_connection: ssh
```

And `inventory/group_vars/lab_vms.yml` carries the rest:

```yaml
---
ansible_user: ansible
ansible_ssh_common_args: >-
  -F {{ lookup('env', 'HOME') }}/.cache/dsoxlab/ansible-training/ssh_config
  -o StrictHostKeyChecking=no
  -o UserKnownHostsFile=/dev/null
ansible_ssh_private_key_file: "{{ inventory_dir }}/../../../../ssh/id_ed25519"
ansible_python_interpreter: /usr/bin/python3.9
```

🔍 **Critical observation**: there is **no `ansible_host`**, hence **no IP**. Ansible connects to the name `web1.lab`, and it is **OpenSSH** that resolves it via the `ssh_config` generated by dsoxlab:

```text
Host web1.lab
  HostName 10.10.20.12
  User ansible
  IdentityFile .../ssh/id_ed25519
```

The libvirt domain name is **also** the name of the `Host` block: what the plugin reports therefore connects without any translation table. Coding the IPs here would make them wrong at the first `dsoxlab provision`.

## 📚 Exercise 5: combine the two inventories

```bash
ansible-inventory -i inventory/ --graph 2>/dev/null | head -20
```

Ansible reads **the whole folder** `inventory/` (the `*.yml` in alphabetical order, plus `group_vars/` and `host_vars/`) and merges. Result: the **dynamic list** from libvirt, plus the **SSH parameters** from the static side.

```bash
ansible web1.lab -i inventory/ -m ansible.builtin.ping
```

Output:

```text
web1.lab | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
```

🔍 **Observation**: Ansible uses SSH (thanks to the `host_vars/` override) and the address from the `ssh_config`, but it is the libvirt plugin that added `web1.lab` to the inventory in the first place.

## 📚 Exercise 6: demonstrate the dynamic side

This is where the **magic** happens. The inventory reflects the state **at run time**, with no cache by default:

```bash
ansible-inventory -i inventory/ --graph state_running 2>/dev/null   # web2.lab is there
virsh -c qemu:///system shutdown web2.lab
ansible-inventory -i inventory/ --graph state_running 2>/dev/null   # web2.lab has disappeared
ansible-inventory -i inventory/ --graph state_shutoff 2>/dev/null   # it is here
```

Without touching a single line of `01-libvirt.yml`, the VM changed group. Restore the initial state:

```bash
virsh -c qemu:///system start web2.lab
```

🔍 **Observation**: this is the **promise** of dynamic inventories. Any VM created by Terraform, Packer, `virt-install` or OpenStack is **automatically** available to Ansible, and any VM shut off drops out of the targets on its own. Zero inventory maintenance.

## 🔍 Observations to note

- **Idempotence**: a second run of your solution must show `changed=0`
  everywhere in the `PLAY RECAP`. This is the mechanical signal of a playbook
  that follows best practices.
- **Explicit FQCN**: always prefer `ansible.builtin.<module>` (or the
  appropriate collection) over the short name: `ansible-lint --profile
  production` checks it.
- **Never a hardcoded IP**: the connection goes through the `ssh_config`
  generated by dsoxlab. This is true in every lab of this training.
- **Isolated reset**: `dsoxlab clean inventaires-dynamique-kvm` cleanly
  uninstalls what the solution set down, to replay the scenario.

## 💡 Going further

- **Filter on the network rather than on a list of names**: the allowlist of
  `lab_vms` remains a list to maintain. The 4 lab VMs have this in common that
  they are on the libvirt network `lab-ansible`, and this information is in
  `xml_desc`. A group `lab_vms: "'lab-ansible' in xml_desc"` would survive the
  addition of a 5th VM. Weigh the pros and cons: less maintenance, but a filter
  by XML substring is less readable and less predictable than an explicit list,
  which matters when a false positive writes on a personal machine.
- **Add a `compose:` Jinja** to compute `ansible_host` from
  `interface_addresses` (useful **only** if the QEMU agent is installed).
- **Install `qemu-guest-agent`** in a VM, then compare `guest_info` before and
  after: this is the difference between a plugin that guesses and a plugin that
  knows.
- **Cache plugin** (`ansible.cfg`) to avoid re-querying libvirt on every
  command, and measure what you lose in freshness.
- **Compare with Proxmox**: `community.general.proxmox` returns VM + IP in a
  single API call.
