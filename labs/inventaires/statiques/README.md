The inventory is the **starting point** of everything Ansible does. Without it,
no playbook knows which machines to act on. The other labs of this section
handed you an inventory that was already written; here, you build it yourself.
This is the official EX294 objective "Create static host inventory files", and
the first task of the mock exam.

### What a static inventory declares

A static inventory is a simple file (YAML or INI) that answers three questions:

1. **Which hosts?** The list of managed machines.
2. **Grouped how?** **Groups** (`webservers`, `dbservers`), and **parent
   groups** that aggregate other groups via `children`.
3. **With which variables?** **Group variables** that apply to all the hosts of
   a group, and are inherited by child groups.

### YAML format

This is the modern format, the one this whole repo uses. Its structure always
follows the same skeleton:

```yaml
---
all:
  vars:                     # variables common to ALL hosts
    ansible_user: ansible
  children:                 # groups live under `children`
    webservers:
      hosts:
        web1.lab:
        web2.lab:
      vars:                 # variable specific to the webservers group
        web_role: frontend
    dbservers:
      hosts:
        db1.lab:
    datacenter:             # PARENT group: no host of its own
      children:             # it aggregates two groups
        webservers:
        dbservers:
      vars:                 # inherited by web1, web2 AND db1
        site: paris
```

Points to watch:

- A host is declared as `web1.lab:` (trailing colon), **not** `- web1.lab`. The
  list form is reserved for other contexts.
- A **parent group** (`datacenter`) does not contain a `hosts:`, it contains
  `children:`. Its variables flow down to the hosts of its children.
- **Group variables** can live inline under `vars:` (as above) or in a
  `group_vars/<group-name>.yml` folder next to the inventory. Both are
  equivalent; this lab uses the inline form.

### The same inventory in INI format

The INI format is still very common, including at the exam. The same content is
written:

```ini
[webservers]
web1.lab
web2.lab

[dbservers]
db1.lab

[datacenter:children]
webservers
dbservers

[webservers:vars]
web_role=frontend

[datacenter:vars]
site=paris
```

A parent group is declared `[<parent>:children]`, group variables
`[<group>:vars]`. This lab requires **YAML** (the standard of the repo and of
the SSH resolution below), but knowing how to translate from one to the other
is expected at EX294.

### No hardcoded IPs: the connection goes through the ssh_config

The addresses of the VMs are assigned dynamically by Terraform: freezing them
in the inventory would make them wrong at the first reprovisioning. So we
delegate resolution to the `ssh_config` generated by dsoxlab, passing it to
Ansible through `ansible_ssh_common_args`. The connection user is **`ansible`**,
the service account dsoxlab provisions on the VMs (SSH key + passwordless sudo);
`student` stays the human account on the control node. This connection block is
provided to you in the challenge: your learning work is about the groups and the
variables.

### Prove your inventory, do not assume it

An inventory that "looks correct" is not enough. You ask Ansible what it
resolves:

```bash
# The resolved structure: groups, children, hosts.
ansible-inventory -i inventory/hosts.yml --graph
ansible-inventory -i inventory/hosts.yml --list

# The effective variables of a host (merged group_vars).
ansible-inventory -i inventory/hosts.yml --host web1.lab

# The ultimate proof: reaching exactly the right hosts.
ansible webservers -i inventory/hosts.yml -m ansible.builtin.ping   # web1 + web2
ansible dbservers  -i inventory/hosts.yml -m ansible.builtin.ping   # db1
ansible datacenter -i inventory/hosts.yml -m ansible.builtin.ping   # all 3
```

If `webservers` reaches db1, or if `datacenter` reaches only two hosts out of
three, the inventory is wrong, even if it reads well. This is exactly what the
tests of this lab check.
